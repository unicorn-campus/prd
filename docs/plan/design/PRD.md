## 2. 개요

### 2.1 배경

- 한국소비자원 조사 기준, 통신 멤버십 가입자 1인당 연평균 지급 포인트 8만 1,452점 중 40.7%만 쓰이고  
  59.3%(약 4만 8,297점)는 유효기간(포인트를 쓸 수 있는 기한)을 넘겨 사라짐(출처: 시장조사.md)
- 문제검증 인터뷰(5명) 결과, "포인트·혜택이 사라진 뒤에야 손해를 안다"는 문제가 평균 중요도 4.2점으로  
  가장 심각하게 확인됨(출처: 문제가설.md)
- 기존 통신사 멤버십 앱은 "조회"만 제공하고 "먼저 챙겨주는 안내"가 없어, 고객이 매번 직접 여러 화면을  
  뒤져야 하고, 결제 대기줄처럼 짧은 시간(8초~40초 탐색 후 포기)에는 확인 자체를 포기함(출처: 문제가설.md)
- 소멸 예정 안내가 광고성 문자와 겉모습이 비슷해, 중요한 안내마저 광고로 여기고 무시하는 패턴이 매달  
  반복됨(출처: 문제가설.md)

### 2.2 제품 한 줄 정의

- 여러 화면을 뒤지지 않아도, 앱이 먼저 내 포인트·등급·혜택과 놓치고 있는 혜택까지 한 화면에 모아  
  보여주고 사라지기 전에 알려주어 손해를 막아주는 통신사 멤버십 부가 기능임
- 비유: 은행 앱이 잔액 부족이나 이상 결제를 먼저 알려주듯, 이 서비스는 포인트·혜택이 사라지기 전에  
  먼저 알려주는 "멤버십 버전 알림 도우미"임(출처: 비즈니스모델.md 고유 가치 제안)

### 2.3 목적

- 고객이 포인트·등급·혜택을 한 화면에서 손쉽게 확인하고, 놓치던 혜택을 앱이 먼저 챙겨주게 하여  
  멤버십 이용 만족도와 앱 재방문을 높임(출처: AGENTS.md 팀 목표)
- 포인트 소멸로 인한 실제 금전 손해를 줄이고, 광고로 오인되던 알림의 신뢰를 회복해 고객이 다시 알림을  
  믿고 확인하게 만듦(출처: 비즈니스가치.md 고객측면 가치1·2·3)

## 3. 문제 정의 및 목표

### 3.1 핵심 문제

문제가설.md에서 5명 인터뷰로 검증된 핵심 문제 3개를 아래와 같이 정의함(괄호 안은 검증 우선순위).

| 문제 | 현재 대안 | 한계 |
|-----|---------|-----|
| 문제1(2순위). 결제 순간 정보가 여러 화면에 흩어져 실시간 확인에 실패함(평균 중요도 3.6점) | 통신사 자체 멤버십 앱에서 화면을 직접 넘겨가며 찾음 | 포인트·등급·혜택이 서로 다른 원장(정보를 저장하는 원본 장부)에 나뉘어 있어 통합 조회 체계가 없음. 짧은 시간에 다 못 찾고 포기함 |
| 문제2(1순위). 포인트·혜택의 유효기간 소멸을 뒤늦게 인지함(평균 중요도 4.2점, 최고) | 소멸 예정 문자·알림 | 소멸 안내와 광고성 메시지가 문구·형식상 구분되지 않아 광고로 여겨 무시함. 매달 반복되며 실제 금전 손해로 이어짐 |
| 문제3(3순위). 매장 현장에서 조건 불일치로 혜택을 놓치거나 거절당함(평균 중요도 3.4점) | 매장 도착 후 직접 확인, 고객센터 문의 | 앱 정보와 매장 결제 단말기 사이 실시간 연동이 없고 정산 주기 단위로만 갱신됨. 등급 관리에 적극적인 일부 고객에 집중된 불편임 |

> 문제3은 검증 결과 3순위이며 특정 고객층에 집중되어, 이번 MVP에서는 직접 해결 대상에서 제외함  
> (근거: 비즈니스가치.md, 유저플로우매핑.md — 자세한 범위는 [5. 범위](#5-범위) 참조)

### 3.2 목표 (Goals)

- G1. 포인트·등급·혜택을 확인하는 데 필요한 화면 전환 횟수를 여러 번에서 한 번(한 장 카드)으로 줄임  
  (문제1 대응, 비즈니스가치.md 고객가치2 연계)
- G2. 소멸 예정 포인트·혜택을 사라지기 전에 미리 알려, 광고와 구분되는 신뢰 알림으로 실제 손해를 줄임  
  (문제2 대응, 비즈니스가치.md 고객가치1·3 연계)
- G3. 앱을 다시 찾는 주간 재방문 이용자 비율을 높이고, 통신사 해지를 줄임  
  (AGENTS.md 목표 및 비즈니스가치.md 회사가치1·2 연계)

### 3.3 비목표 (Non-Goals)

- 매장 현장 조건 사전 확인(문제3)은 이번 MVP에서 직접 해결하지 않음(Out of Scope, [5. 범위](#5-범위) 참조)
- 포인트 결제·전환·가족 공유·기부 등 포인트 자체를 옮기거나 쓰는 기능은 다루지 않음(범용 포인트  
  통합 서비스 영역, 출처: 솔루션평가.md 보류·폐기 후보)
- 사람이 직접 상담하는 고객센터 전화 흐름은 이번 범위에 포함하지 않음(출처: 고객분석.md 제외 사유)
- 사람의 이용 패턴을 학습하는 인공지능(AI) 추천 모델은 이번 MVP에서 만들지 않음(단순 규칙 방식으로  
  대체, 출처: 06 이벤트 스토밍 주석, 솔루션평가.md)

- 얼리어답터(가장 먼저 새 기능을 반기고 적극적으로 써 보는 초기 사용자)는 세그먼트4(등급 관리형 고객)임

  가장 먼저 반기고 입소문(가족 간 추천)을 낼 가능성이 큰 집단임(출처: 비즈니스모델.md Early Adopters)

## 5. 범위

범위는 솔루션평가.md의 실현 가능성(지금 우리 힘으로 만들기 쉬운 정도) 평가와 정합성을 유지함.  
평가 결과 "포인트·등급·혜택 한 장 카드"와 "신호등 뱃지"는 즉시 실행(사업가치·실현가능성 모두 높음),  
"개인화 추천"은 전략 투자(사업가치 높음·실현가능성 낮음)로 분류되어 단순 규칙 방식으로 시작함.

### 5.1 In Scope (이번 MVP 포함)

- 통합 혜택 카드 조회: 포인트·등급·혜택을 한 화면 카드로 통합 표시(유저플로우 01, 솔루션1)
- 잠금화면·홈화면 위젯 확인: 앱을 열지 않아도 핵심 정보를 요약 표시(유저플로우 02, 솔루션1)  
  > 위젯(widget): 앱을 열지 않아도 홈 화면·잠금화면에 정보를 보여주는 작은 정보창임
- 결제 현장 혜택 즉시 확인: 지금 있는 매장에서 쓸 수 있는 혜택을 즉시 조회(유저플로우 03, 솔루션1)
- 소멸 임박 신호등 알림: 소멸까지 남은 일수를 신호등 색·숫자로 사전 알림(유저플로우 04, 솔루션2)
- 놓친 혜택 복구 안내: 지난번 놓친 혜택을 요약해 다시 알려주고 알림 설정을 조정(유저플로우 05, 솔루션2)
- 개인화 혜택 추천: 자사 이용실적 기반 단순 규칙으로 지금 맞는 혜택 1건 추천(유저플로우 06, 솔루션3)  
  > 인공지능(AI) 추천 모델은 만들지 않고, "최근 30일 최다 결제 업종 1개" 같은 단순 규칙으로만 시작함
- 혜택 추천·알림 동의 설정: 개인정보·위치정보·알림수신을 항목별로 나눠 동의받는 사전 절차  
  (유저플로우 08, 규제 대응 공통 사전 단계)

### 5.2 Out of Scope (이번 MVP 제외)

- 매장 방문 전 조건 사전 확인(유저플로우 07): 문제3이 검증 3순위이고 특정 고객층에 집중되어 제외함.  
  설계 참고용으로만 유지하며 MVP 이후 검토함(출처: 유저플로우매핑.md, 비즈니스가치.md)
- 실시간 위치 기반 인근 유사 혜택 매장 재탐색: MVP 범위 초과로 "혜택 없음 안내"까지만 제공(출처: 03 주석)
- 인공지능(AI) 학습 기반 추천 모델: 이번에는 단순 규칙 방식만 적용(출처: 06 주석, 솔루션평가.md)
- 매장 근처 접근 시 자동으로 알려주는 위치 기반 알림: 사업가치·실현가능성 모두 최하위로 폐기  
  (출처: 솔루션평가.md 후보3, 0표)
- 포인트 결제·전환·가족 공유·기부: 범용 포인트 통합 서비스 영역으로 제외(출처: 솔루션평가.md)
- 외부 기관 정보를 끌어오는 마이데이터 연계: 자사 데이터베이스 조회로 단순화(출처: 유저플로우매핑.md,  
  통신비밀보호법 근거 오귀속 해소)
- 미성년 고객 법정대리인 동의 절차: 통신 회선 가입 시점에 성인 인증이 완료된 것으로 가정하여 제외  
  (출처: 08 주석)

## 6. 사용자 시나리오

이벤트 스토밍 8개 유저플로우를 사용자 관점의 핵심 흐름으로 서술함. 각 시나리오 끝에 대응하는 기능  
요구사항(FR) ID를 표기하여 [7. 기능 요구사항](#7-기능-요구사항)과 연결함.

### 시나리오 1. 통합 혜택 카드 한눈에 조회 (유저플로우 01)

1. 고객이 앱에 진입해 통합 혜택 카드 조회를 요청함
2. 앱이 포인트 잔액·등급·보유 혜택·포인트 유효기간·소멸예정일을 한 장 카드로 표시함
3. 정보를 불러오지 못하면 실패 안내와 다시 시도 경로를 보여주고, 재시도 시 카드를 정상 표시함
   - 관련 FR: FR-01, FR-02, FR-03

### 시나리오 2. 잠금화면·홈화면 위젯 확인 (유저플로우 02)

1. 고객이 휴대폰 화면을 켜면 잠금화면·홈화면 위젯에 포인트 요약·등급·소멸 D-day가 표시됨
   - D-day: 소멸까지 남은 날짜를 "D-30"(30일 남음)처럼 표시하는 방식임
2. 정보 갱신에 실패하면 이전 값과 "마지막 갱신 시각"을 함께 보여주어 오해를 막음
3. 위젯을 누르면 앱의 통합 혜택 카드 화면(시나리오 1)으로 바로 이동함
   - 관련 FR: FR-04, FR-05, FR-06

### 시나리오 3. 결제 현장 혜택 즉시 확인 (유저플로우 03)

1. 고객이 매장에서 지금 쓸 수 있는 혜택 조회를 요청함
2. 제휴 매장이면 사용 가능 혜택을, 혜택이 없으면 없음 안내를 즉시 보여줌
3. 위치 권한이 꺼져 있으면 매장명을 직접 검색하는 대체 경로를 제공함
4. 결제 시점에 혜택 사용을 확정하며, 같은 혜택이 두 번 차감되지 않도록 막음
   - 관련 FR: FR-07, FR-08, FR-09

### 시나리오 4. 소멸 임박 신호등 알림 (유저플로우 04)

1. 매일 자정 기준으로 소멸 예정 포인트·혜택 대상을 일괄 계산함
2. 신유형 상품권 표준약관에 따라 소멸 30일 전(1차)·7일 전(2차)에 신호등 뱃지 알림을 발송함
3. 알림은 광고와 구분되는 전용 발신 방식("혜택 도우미" 표시)으로 보내 신뢰를 확보함
4. 고객이 알림을 확인하지 않으면 소멸 3일 전·1일 전에 앱 안에서 신호등 뱃지를 다시 노출함
   - 관련 FR: FR-10, FR-11, FR-12

### 시나리오 5. 놓친 혜택 복구 안내 (유저플로우 05)

1. 오랫동안 앱을 열지 않은 고객을 자동으로 찾아 놓친 혜택 요약 알림을 보냄
2. 고객이 오랜만에 앱에 재접속하면 지난번 놓친 혜택 요약 카드를 보여줌
3. 이력 조회에 실패하면 기본 포인트·등급·혜택 카드만이라도 보여줌(핵심 정보 우선 제공)
4. 고객 반응에 따라 알림을 유지·강화하거나, 무관심 고객은 알림 빈도를 완화함
   - 관련 FR: FR-13, FR-14, FR-15

### 시나리오 6. 개인화 혜택 추천 확인 (유저플로우 06)

1. 동의한 고객에 한해 자사 결제 업종·이용 빈도·등급 데이터를 조회함
2. 통화·문자 등 통신 내용은 절대 활용하지 않고, 이용 실적만으로 단순 규칙 매칭을 수행함
3. "지금 나에게 딱 맞는 혜택" 1건을 추천 사유·예상 절감액과 함께 알림·표시함
4. 조회 실패 시 기본 혜택 안내로 대체하고, 동의를 철회하면 개인화 추천을 즉시 중단함
   - 관련 FR: FR-16, FR-17, FR-20

### 시나리오 7. 매장 방문 전 조건 확인 (유저플로우 07, MVP 이후 검토)

1. 고객이 방문 예정 매장의 혜택 사용 조건(등급 제한·최소 구매 금액·사용 가능 시간)을 조회함
2. 조건 충족 여부를 안내하고, 미충족 시 인근 대안 매장을 안내함
   - 관련 FR: FR-18 (우선순위 W = 이번엔 제외, [5. 범위](#5-범위) 참조)

### 시나리오 8. 혜택 추천·알림 동의 설정 (유저플로우 08)

1. 고객이 최초 진입 시 개인정보(이용실적)·위치정보·알림수신을 항목별로 나눠 동의함
2. 전체 동의·부분 동의·미동의를 선택할 수 있고, 동의 범위 안에서만 기능이 활성화됨
3. 동의 상태가 바뀌면 관련 서비스 전체에 전파되어, 철회 시 개인화 추천이 즉시 멈춤
4. 설정 화면에서 언제든 다시 동의하거나 철회할 수 있음
   - 관련 FR: FR-19, FR-20, FR-21

## 8. 비기능 요구사항

성능·가용성·보안·확장성 4개 영역을 모두 포함하며, 모두 측정 가능한 목표치와 측정 방법을 함께 정의함.  
이 목표치는 이후 유저스토리 단계에서 재정의하지 않는 단일 기준(Source of Truth, 하나의 원본 기준)임.  
"빠르게·안전하게" 같은 모호한 표현은 사용하지 않음.

> 용어: p95(응답이 빠른 순서로 줄 세웠을 때 95번째 값으로, 대부분 사용자가 겪는 속도) /  
> 서킷브레이커(circuit breaker, 연결된 시스템이 고장 나면 자동으로 잠시 끊고 대체 화면을 보여주는 안전장치) /  
> 열화대응(일부 기능이 안 될 때 핵심 정보만이라도 보여주는 대응)

> 멱등성(idempotency): 같은 요청을 여러 번 보내도 결과가 한 번 한 것과 같도록 만들어 중복을 막는 성질임

### 8.4 확장성 (이용자가 늘어도 견디는가)

| 요구사항(목표치) | 측정 방법 | 근거 |
|----------------|----------|------|
| 동시 조회 초당 5,000건 이상 안정 처리 | 부하 테스트(초당 요청 수 대비 응답시간·오류율) | 대규모 조회 트래픽 이력 |
| 조회(실시간)와 알림·추천(배치)의 서버 확장을 서로 독립적으로 조절 | 서비스별 자동 확장(오토스케일링) 지표 분리 관측 | 부하 변동에 따른 서비스 분리 |
| 위젯 주기 갱신·알림 발송 직후 몰림에 지터·큐 완충 적용, 발송 지연 목표 이내 유지 | 대기열 적체량·발송 지연 모니터링 | 트래픽 스파이크 대응 정책 |

> 지터(jitter): 요청이 한꺼번에 몰리지 않도록 시점을 조금씩 무작위로 흩어주는 기법임 /  
> 큐(queue): 밀려드는 작업을 줄 세워 순서대로 처리하는 대기열임 /  
> 오토스케일링(auto scaling): 이용량이 늘면 서버를 자동으로 늘리는 기능임

## 9. UX 원칙

> UX(User Experience, 사용자 경험): 고객이 서비스를 쓰면서 느끼는 전반적인 편함·만족을 뜻함

- 짧은 시간 안에 끝나야 함: 결제 대기줄처럼 급한 상황을 고려해 핵심 정보는 한 화면(한 장 카드)에서  
  바로 확인되게 함(문제1 대응)
- 앱을 열지 않아도 보이게 함: 잠금화면·홈화면 위젯으로 핵심 정보를 3초 이내에 확인하게 함
- 알림은 믿을 수 있게 함: 소멸·놓침·추천 알림을 광고와 뚜렷이 구분되는 전용 방식으로 보내, 광고로  
  오인해 무시하던 문제를 해소함(문제2 대응)
- 왜 보여주는지 함께 설명함: 추천 혜택에는 추천 사유·예상 절감액을 함께 보여줘 납득할 수 있게 함
- 실패해도 막히지 않게 함: 조회·연동이 실패해도 재시도 경로·대체 경로·핵심 정보 우선 표시로 흐름이  
  끊기지 않게 함(감정 저점 단계 보완)
- 고객이 통제하게 함: 동의·알림 빈도를 고객이 항목별로 켜고 끌 수 있게 해 신뢰를 지킴

## 10. 성공 지표

비즈니스가치.md의 측정 지표 및 비즈니스모델.md의 핵심 지표(AARRR)와 연계함. AARRR은 획득·활성화·  
유지·매출·추천 5단계로 서비스 성장을 점검하는 방법임.

| 지표 | 정의 | 목표치 | 연계 근거 |
|-----|-----|-------|----------|
| 확인까지 화면 전환 횟수 | 포인트·혜택 확인에 필요한 화면 넘김 수 | 여러 번 → 1회(한 장 카드) | 비즈니스가치 고객가치2 |
| 포인트 소멸 방지율 | 유효기간 안에 쓰거나 지켜낸 포인트 비율 | 현재 소멸률 59.3% 대비 소멸 방지율 상향 | 비즈니스가치 고객가치1 |
| 알림 후 실제 확인율 | 혜택 도우미 알림 발송 대비 실제 확인 비율 | 40% 이상 | 비즈니스가치 고객가치3 / AARRR 추천 |
| 첫 사용률(활성화) | 안내 후 한 장 카드 화면을 최초 확인한 비율 | 70% 이상 | AARRR 활성화 |
| 주간 재방문 이용자 비율 | 일주일 안에 앱을 다시 연 고객 비율 | 기존 대비 +15%p | 비즈니스가치 회사가치1 / AARRR 유지 |
| 해지율 감소폭 | 통신사를 그만두는 고객 비율의 감소 정도 | 1년차 0.3%p → 3년차 1.0%p 감소 | 비즈니스가치 회사가치2 / AARRR 유지 |
| 신규 기능 인지·사용 도달률 | 최우선 세그먼트 대비 실제 기능 사용자 비율 | 1년차 15% → 2년차 45% → 3년차 65% | 비즈니스모델 AARRR 획득 |

> %p(퍼센트포인트): 두 비율의 차이를 나타내는 단위임(예: 40%에서 55%로 오르면 15%p 증가)

## 12. 리스크 및 가정

### 12.1 가정

- 통신 회선 가입 시점에 성인 인증이 완료된 것으로 가정함(미성년 법정대리인 동의 절차 제외 근거)
- 자사에 포인트·등급·혜택 원부와 캐시(임시 저장본)를 이원화한 조회 기반이 갖춰져 있다고 가정함
- '놓친 혜택' 판별과 개인화 추천의 근거가 되는 이용내역·제휴혜택 데이터를 안정적으로 확보할 수 있고,  
  소멸·혜택 정보가 실시간에 가깝게 최신 상태로 유지된다고 가정함. 이 전제가 무너지면 FR-13~FR-16과  
  성공지표 전반이 무효가 되므로, 유저스토리 착수 전 데이터 확보 가능성·최신성을 반드시 확인함
- 제휴 매장 시스템·운영체제 위젯·푸시 알림 발송 서비스와 연동이 가능하다고 가정함([13. 의존성](#13-의존성))
- 위치 기반 기능은 고객이 위치정보 제공에 동의한 경우에만 작동한다고 가정함
- 시장 규모(SOM 등) 추정치는 사내 가입자 데이터로 재검증이 필요함(출처: 시장조사.md 주의 문구)

### 12.2 리스크

도메인 특성상 규제 위반은 서비스 신뢰와 직결되므로, 실패 시나리오를 함께 명시함.

| 리스크 | 영향(실패 시나리오) | 대응 방안 |
|-------|-------------------|----------|
| 포인트 유효기간·소멸 사전 고지 누락 | D-30·D-7 고지가 누락되면 신유형 상품권 표준약관·전자상거래 소비자보호법 위반으로 공정거래위원회 시정 대상이 됨(불공정 약관 85건 시정 사례 참고) | 유효기간·소멸예정일 상시 표시(FR-02)와 D-30·D-7 발송률 100%(8.3)를 필수 목표로 관리 |

| 통신 내용(통화·문자) 활용 | 이용 실적을 넘어 통신 내용을 쓰면 통신비밀보호법 위반이 됨 | 자사 이용 실적만 활용, 통신 내용 활용 0건(8.3), 외부 마이데이터 연계 제외 |

| 캐시-원부 데이터 불일치 | 임시 저장본이 오래돼 잘못된 포인트가 표시되면 고객 신뢰가 하락함 | 원부 신뢰 원칙·캐시 무효화(FR-01), 혜택 사용 시 캐시 무효화 트리거 |

> 콜드 스타트(cold start): 이용 이력이 거의 없는 초기 상태라 추천 정확도가 낮아지는 문제임

| 캐시 무효화 | 정보가 바뀌었을 때 임시 저장본을 지워 최신값을 다시 불러오게 하는 것임 |
| 위젯(widget) | 앱을 열지 않아도 홈 화면·잠금화면에 정보를 보여주는 작은 정보창임 |
| 딥링크(deep link) | 위젯·알림을 누르면 앱의 특정 화면으로 바로 들어가게 하는 바로가기 링크임 |
| D-day | 소멸까지 남은 날짜를 "D-30"(30일 남음)처럼 표시하는 방식임 |
| 배치(batch) | 정해진 시간에 대량의 작업을 한꺼번에 처리하는 방식임 |
| 낙관적 락 | 동시에 같은 혜택을 두 번 쓰지 못하게 막는 중복 방지 장치임 |
| 멱등성 | 같은 요청을 여러 번 보내도 결과가 한 번 한 것과 같게 만들어 중복을 막는 성질임 |
| 서킷브레이커 | 연결된 시스템이 고장 나면 자동으로 잠시 끊고 대체 화면을 보여주는 안전장치임 |
| 타임아웃 | 정해진 시간 안에 응답이 없으면 기다림을 멈추는 것임 |
| 열화대응 | 일부 기능이 안 될 때 핵심 정보만이라도 보여주는 대응임 |
| 지터(jitter) | 요청이 한꺼번에 몰리지 않도록 시점을 조금씩 무작위로 흩어주는 기법임 |
| 큐(queue) | 밀려드는 작업을 줄 세워 순서대로 처리하는 대기열임 |
| 오토스케일링 | 이용량이 늘면 서버를 자동으로 늘리는 기능임 |
| p95 | 응답이 빠른 순서로 줄 세웠을 때 95번째 값으로, 대부분 사용자가 겪는 속도임 |
| MoSCoW | 기능 중요도를 M(반드시)·S(권장)·C(선택)·W(이번엔 제외) 네 단계로 나누는 방법임 |
| JTBD | 고객이 진짜 이루고 싶은 일(목표)을 찾는 방법임(Jobs To Be Done) |
| AARRR | 획득·활성화·유지·매출·추천 5단계로 서비스 성장을 점검하는 방법임 |
| SOM | 1~2년 안에 실제로 확보를 노려볼 수 있는 시장 규모임(Serviceable Obtainable Market) |
| %p(퍼센트포인트) | 두 비율의 차이를 나타내는 단위임 |
| 콜드 스타트 | 이용 이력이 거의 없는 초기 상태라 추천 정확도가 낮아지는 문제임 |

| 02 | 잠금화면·홈위젯 확인 | 포인트·등급·혜택 한 장 카드 | MVP |

## 2. 유저플로우-여정 단계 매핑

유저저니맵.md의 감정 곡선 기준, 감정 레벨 2 이하 단계(2. 정보 탐색=2점, 3. 확인 또는 포기=2점,  
5. 뒤늦은 손해 인지=1점, 6. 체념 및 반복=2점)에 대응하는 유저플로우에는 에러처리·피드백·안내메시지·  
복구경로 등 UX 보완 이벤트를 반영함(각 puml 파일 내 alt 분기로 표현).

| 유저플로우 | 고객 여정 단계 | 감정 레벨 | 비고 |
|-----------|-------------|----------|------|
| 01. 통합 혜택 카드 조회 | 2. 정보 탐색(앱 헤매기) | 2 | 여러 화면 탐색을 한 화면으로 대체함. 조회 실패 시 재조회 경로(UX 보완) 추가함 |
| 02. 잠금화면·홈위젯 확인 | 1. 혜택 필요 인식 | 3 | 앱을 열지 않아도 핵심 정보 노출, 3초 이내 확인을 목표로 함 |
| 03. 결제 현장 혜택 즉시 확인 | 3. 확인 또는 포기 | 2 | 혜택 없음·조회 지연 시 안내 및 대안 매장 제시(UX 보완)로 포기 최소화함 |
| 04. 소멸 임박 신호등 알림 | 5. 뒤늦은 손해 인지 | 1 | 손해 인지 전 사전 알림으로 감정 저점을 예방함. 미확인 시 인앱 재노출(UX 보완) 추가 |
| 05. 놓친 혜택 복구 안내 | 6. 체념 및 반복 | 2 | 재접속 시 놓친 내역 요약과 알림 빈도 재조정(UX 보완)으로 신뢰 회복을 유도함 |
| 06. 개인화 혜택 추천 확인 | 1. 혜택 필요 인식 | 3 | 고객이 찾기 전에 앱이 먼저 추천함(자사 결제업종·이용빈도 데이터 기반 규칙 매칭, MVP는 추천모델 미적용) |
| 07. 매장 방문 전 조건 확인 | 4. 혜택 사용 시도 | 3 | **MVP 이후 검토**. 현장 거절 상황을 방지하기 위한 사전 조건 안내이나, 관련 문제(매장 현장 조건 불일치)가  
  문제가설 검증 결과 3순위로 확인되어 비즈니스가치.md에서 이미 제외 결정됨(피오 리뷰 반영). 설계 참고용으로만 유지함 |
| 08. 혜택추천 동의 설정 | 사전 단계(전 단계 공통) | - | 개인정보보호법·통신비밀보호법 대응을 위한 최초 진입 시 1회성 절차 |

## 3. 규제 반영 사항

시장조사.md의 규제/법적 환경을 유저플로우 내 정책/규칙으로 반영함.

| 규제 | 반영 유저플로우 | 반영 내용 |
|------|----------------|----------|
| 개인정보보호법 | 08. 혜택추천 동의 설정 | 이용목적·수집항목(이용실적)을 사전 고지하고 동의를 받은 뒤에만 개인화 기능 활성화함 |
| 위치정보의 보호 및 이용 등에 관한 법률 | 08. 혜택추천 동의 설정 | 위치정보는 개인정보(이용실적) 동의와 별도 항목으로 분리해 고지·동의를 받음(인사이트 리뷰 반영 -  
  기존에는 위치정보가 개인정보 동의에 뭉뚱그려져 있었음) |
| 전자상거래 등에서의 소비자보호에 관한 법률 | 01. 통합 혜택 카드 조회, 04. 소멸 임박 신호등 알림 | 포인트 유효기간·소멸 예정일을 화면과 알림에 상시 표시함 |
| 신유형 상품권 표준약관 | 04. 소멸 임박 신호등 알림 | 소멸 전 사전고지 의무를 최소기준(소멸 30일 전 1차, 7일 전 2차)으로 구체화해 배치 계산·알림 발송  
  정책/규칙으로 반영함(인사이트 리뷰 반영) |
| 통신비밀보호법 | 06. 개인화 혜택 추천 확인 | 자사 멤버십조회서비스가 보유한 결제업종·이용빈도 데이터만 활용하고 통화·문자 등 통신내용은 활용  
  범위에서 제외함. 기존 설계의 "(E)마이데이터연계시스템"(외부 연계, 신용정보법 대상)은 자사 DB  
  조회로 단순화해 법적 근거 오귀속 문제를 해소함(인사이트 리뷰 반영) |
| (해당 없음 - MVP 범위 제외) | 08. 혜택추천 동의 설정 | 미성년 고객 법정대리인 동의 절차는 MVP 범위에서 제외함. 통신 회선 가입 시점에 이미 성인 인증이  
  완료된 것으로 가정하며, 이는 설계 변경이 아닌 범위 명시 사항임(인사이트 리뷰 반영) |

## 4. 마이크로서비스 정의

### 4.1 정의 절차 (5단계)

1. **이벤트 클러스터링**: 유저플로우 8개에서 발생한 이벤트를 "조회/표시" 성격과 "알림/추천/동의" 성격  
   두 그룹으로 클러스터링함
   - 조회/표시 그룹: 포인트등급혜택카드표시됨, 위젯정보전달됨, 매장혜택조회됨, 사용조건확인됨
   - 알림/추천/동의 그룹: 소멸임박대상산출됨, 인앱배너재노출됨, 추천혜택산출됨, 동의등록됨
2. **Aggregate 식별**: 각 이벤트가 변경하는 엔티티와 생명주기를 파악함
   - 포인트, 등급, 보유혜택(조회/표시 그룹 - 원부 데이터 기준 변경, 조회는 캐시 우선)
   - 소멸예정알림, 추천결과, 이용패턴, 동의이력(알림/추천/동의 그룹 - 배치·이벤트성 변경)
3. **바운디드 컨텍스트 정의**: Aggregate와 정책/규칙을 묶어 두 개의 경계로 정리함
   - 멤버십조회 컨텍스트: 포인트·등급·혜택 조회 정책(캐시 우선 조회, 소비자보호법 고지)
   - 혜택알림추천 컨텍스트: 소멸임박 산정·발송 정책, 개인화 추천 정책, 동의관리 정책
4. **컨텍스트 맵핑**: 멤버십조회 컨텍스트가 혜택알림추천 컨텍스트에 포인트·등급·이용내역 데이터를  
   제공(Published Language)하고, 혜택알림추천 컨텍스트는 동의 여부를 조회해 알림·추천 실행 여부를  
   결정하는 관계로 정의함
5. **분할/병합 결정**: MVP 단계 병합 전략에 따라 아래 2개 마이크로서비스로 병합함

### 4.2 분할/병합 근거

| 판단 | 근거 |
|-----|------|
| 조회 기능은 별도 서비스로 분리 | 조회 트래픽 부하 패턴(초당 다건 조회)이 알림·추천 배치 처리와 스케일링 패턴이 다름(분할 기준: 부하 변동) |
| 소멸알림·개인화추천·동의관리는 하나로 병합 | 코드 규모가 각각 5,000줄 미만으로 작을 것으로 예상되고(병합 기준: 서비스 크기), 동의 여부가 알림·추천  
  실행의 선행 조건으로 호출 빈도가 높아(병합 기준: 호출 빈도) 트랜잭션 경계상 함께 관리하는 것이 운영에  
  유리함. 향후 개인화 추천 로직이 고도화(AI 모델 적용 등)되어 변경 빈도·기술 스택이 달라지면 별도  
  서비스로 분리 검토함(핵심솔루션.md에서 실현가능성이 가장 낮게 평가된 솔루션이므로 초기에는 규칙  
  기반의 단순한 형태로 병합 운영함) |

### 4.3 마이크로서비스 정의 표

| 서비스명 | 바운디드 컨텍스트 | 관련 유저플로우 | 주요 이벤트 |
|---------|----------------|-------------|-----------|
| 멤버십조회서비스 | 포인트·등급·혜택 조회 | 01, 02, 03, 07 | 포인트등급혜택카드표시됨, 위젯용요약정보전달됨, 매장혜택조회됨, 사용조건확인됨, 보유혜택사용됨 |
| 혜택알림추천서비스 | 소멸임박알림 + 개인화추천 + 동의관리 | 04, 05, 06, 08 | 소멸임박대상산출됨, 인앱신호등뱃지재노출됨, 추천혜택산출됨, 추천혜택사용됨/미사용됨, 동의상태변경됨 |

### 4.4 구현 제약 메모 (리뷰 반영 - 배포/운영 수준)

파이프(DO), 데브백(BE) 리뷰 의견 중 도메인 경계(4.3의 서비스 정의) 자체를 바꾸지 않고, 배포·운영  
수준의 구현 제약으로 반영하는 것이 적절한 항목을 정리함. 이 표의 내용은 설계(HighLevel 아키텍처,  
물리 아키텍처) 단계에서 상세화함.

| 항목 | 메모 | 리뷰 출처 |
|-----|------|----------|
| 배포 단위 분리 | 혜택알림추천서비스 내에서도 04(배치성)와 06/08(API성)은 도메인 경계는 유지하되 배포 단위는  
  분리함(04는 K8s CronJob, 06/08은 Deployment+HPA) | 파이프 |
| 배치워커/API서버 프로세스 분리 | 04(배치워커)와 06(API서버)은 워크로드 특성(처리량 vs 응답속도)이 달라 프로세스 단위로 분리  
  구현함 | 데브백 |
| 캐시 무효화 전략 | 03의 보유혜택사용됨 이벤트를 캐시 무효화 트리거로 사용함. 01에서 캐시-원부 불일치 감지 시  
  원부 데이터를 신뢰하고 캐시를 무효화함 | 데브백, 파이프 |
| 트래픽 스파이크 대응 | 02 위젯 폴링과 04 알림 발송 직후 트래픽이 몰릴 수 있어 지터(jitter, 요청 시점을 무작위로 소폭  
  분산시키는 기법)를 적용하고, 알림 발송은 큐 기반 비동기 처리로 완충함 | 파이프 |
| 외부·내부 연동 회복성 | 03(제휴매장), 06(내부 DB), 07(제휴매장) 연동에 타임아웃(3초)+서킷브레이커(장애 시 자동 차단  
  후 대체 응답) 정책을 적용함(각 puml에 반영) | 파이프 |
| 동의조회 캐시 | 08 동의 상태 조회가 06 호출마다 매번 DB 조회로 이어지면 핫스팟(특정 자원에 요청이 몰리는 현상)  
  위험이 있어 캐시 패턴 적용을 검토함 | 파이프 |

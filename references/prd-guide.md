# PRD 작성 가이드

## 목적

이벤트 스토밍으로 파악된 도메인 흐름과 이전 단계 산출물(비즈니스모델·문제가설·시장조사)을 근거로,
제품 범위와 기능/비기능 요구사항을 확정하는 PRD(Product Requirements Document)를 작성함.
PRD는 "무엇을 만들지·왜 만들지"를 확정하는 문서이며, 유저스토리는 PRD를 "어떻게 구현할지" 실행 단위로 분해함.

## 입력 (이전 단계 산출물)

| 산출물 | 파일 경로 | 활용 방법 |
|--------|----------|----------|
| 이벤트 스토밍 | `docs/plan/think/es/*.puml` | 유저플로우 → 사용자 시나리오, 커맨드 → 기능 요구사항 후보, 정책/규칙 → 비기능 요구사항·제약조건 |
| 비즈니스 모델 | `docs/plan/think/비즈니스모델.md` | Problem/Customer Segments/UVP → 문제 정의·타겟 사용자·목적, Key Metrics → 성공 지표 |
| 문제 가설 | `docs/plan/define/문제가설.md` | 핵심 문제 Top 3 → 문제 정의 근거 |
| 시장 조사 | `docs/plan/define/시장조사.md` | 경쟁 환경·규제 환경 → 범위 설정·비기능 요구사항(컴플라이언스) 근거 |
| 고객 분석 | `docs/plan/define/고객분석.md` | 사용자 세그먼트 → 타겟 사용자·페르소나 |
| 비즈니스 가치 | `docs/plan/define/비즈니스가치.md` | 측정 지표 → 성공 지표(Success Metrics) |

## 출력 (이 단계 산출물)

| 산출물 | 파일 경로 |
|--------|----------|
| PRD | `docs/plan/design/PRD.md` |

## 이벤트스토밍 → PRD 연결

| Event Storming 요소 | PRD 요소 | 설명 |
|-------------------|---------|------|
| **유저플로우(제목)** | 사용자 시나리오 | 제품이 지원해야 하는 핵심 흐름 |
| **이벤트 [이벤트]** | 기능 요구사항 후보 | 시스템이 보장해야 하는 결과 |
| **커맨드 [커맨드]** | 기능 요구사항 상세 | 사용자/시스템이 수행하는 행동 |
| **정책/규칙 [정책/규칙]** | 비기능 요구사항·제약조건 | 성능·보안·컴플라이언스 등 시스템 전반 제약 |
| **참여자 (Actor)** | 타겟 사용자 | PRD의 사용자 세그먼트 정의에 반영 |

> **원칙**: 이벤트 스토밍에서 식별된 모든 유저플로우는 최소 1개 이상의 기능 요구사항(FR)과 연결되어야 함.

## 방법론

### PRD 구성 요소 (14개 섹션)

1. **문서 정보**: 제품명, 버전, 작성일, 상태
2. **개요**: 배경, 제품 한 줄 정의, 목적
3. **문제 정의 및 목표**: 핵심 문제, Goals, Non-Goals
4. **타겟 사용자**: 페르소나, Early Adopter 특성
5. **범위**: In Scope / Out of Scope
6. **사용자 시나리오**: 핵심 플로우별 단계 서술 (이벤트스토밍 유저플로우 기반)
7. **기능 요구사항**: MoSCoW 우선순위 기반 기능 목록 (이벤트스토밍 커맨드/이벤트 기반)
8. **비기능 요구사항**: 성능·가용성·보안·확장성 등 목표치 (정책/규칙 기반)
9. **UX 원칙**: 핵심 UX 방향
10. **성공 지표**: 정량 KPI (비즈니스가치·Key Metrics 연계)
11. **릴리스 계획**: 버전별 범위·일정
12. **리스크 및 가정**: 전제조건, 리스크·대응방안
13. **의존성**: 외부 API/서비스 연동
14. **부록**: 용어집, 참고 산출물

### 비기능 요구사항 작성 원칙

- 비기능 요구사항은 **목표치(threshold)** 로 정의함 (예: "API 응답 2초 이내", "동시 세션 1,000개 이상")
- 이 목표치는 이후 유저스토리 작성 시 스토리별 구현 태스크·수용 기준으로 구체화되는 **단일 출처(Source of Truth)** 역할을 함
- 유저스토리에서 동일 목표치를 재정의하지 않도록, 목표치는 PRD에서만 관리함

## 품질 기준

### 완료 체크리스트

- [ ] 14개 섹션 모두 작성
- [ ] 문제 정의가 `문제가설.md`의 핵심 문제와 연결됨
- [ ] 기능 요구사항이 이벤트스토밍의 커맨드/이벤트와 연결됨
- [ ] 기능 요구사항에 MoSCoW 우선순위 전체 적용
- [ ] 비기능 요구사항에 측정 가능한 목표치 포함 (성능·가용성·보안·확장성)
- [ ] 범위(In/Out Scope) 명확히 구분
- [ ] 성공 지표가 `비즈니스가치.md`의 측정 지표와 연계됨
- [ ] 이벤트스토밍의 모든 유저플로우가 최소 1개 이상 기능 요구사항과 연결됨

### 정량 기준

- 기능 요구사항: 이벤트스토밍 유저플로우 수 이상
- 비기능 요구사항: 성능·가용성·보안·확장성 4개 영역 모두 포함
- 성공 지표: 3개 이상

## 주의사항

- 기능 요구사항은 이벤트스토밍 결과를 근거로 도출하고, 근거 없는 신규 기능 임의 추가 금지
- 비기능 요구사항은 구체적 수치로 작성 (모호한 "빠르게", "안전하게" 표현 금지)
- 비기능 요구사항 목표치는 PRD에서만 정의하고, 후속 유저스토리 단계에서는 재정의하지 않고 구현 태스크로만 구체화
- In Scope/Out of Scope는 이해관계자 간 오해를 막기 위해 명시적으로 나열
- 범위는 실현 가능성(솔루션평가.md)과 정합성 유지

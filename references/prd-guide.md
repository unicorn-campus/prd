# PRD 작성 가이드 (LG유플러스 서비스 기획서 양식)

## 목적

이벤트 스토밍으로 파악된 도메인 흐름과 이전 단계 산출물(비즈니스모델·문제가설·시장조사·고객분석)을 근거로,
LG유플러스 B2C 서비스 기획서 양식의 PRD(Product Requirements Document)를 작성함.
표지 + 13개 섹션 구조로 "무엇을·왜 만들지"를 확정하고, 기존/경쟁사 대비 차별점과 화면·알림·정책·예외까지
빠짐없이 명시하여, 개발·품질(QA)·디자인 담당자가 바로 착수할 수 있게 함.
최종 산출물은 샘플과 동일한 서식의 Word(.docx) 파일임.

## 방법론

### 문서 구성 (표지 + 13개 섹션)

작성 순서·섹션 제목·표 열 구성은 샘플과 동일하게 유지함.

- **표지**: 문서 제목, 교육/샘플 안내 문구, `{제품명} 기획서 V {버전} / {작성일} / {팀명 작성자}` 한 줄
- **1. Revision History**: 버전 이력 (Ver. / Date / Writer / Description / Tag)
- **2. 서비스 기획서 요구사항 체크리스트**: 항목 / 개요 / 필수여부(M·O) / 내용
- **3. 서비스 개요**: 3.1 배경 / 3.2 타겟 / 3.3 서비스 컨셉(한 줄)
- **4. 상품 구성**: 4.1 디바이스 스펙 / 4.2 요금제
- **5. 페르소나**: 5.1 Persona A / 5.2 Persona B (항목 / 내용)
- **6. 정책 비교표 (킬러 폼)**: 기존 vs 경쟁사 vs 신규 3열 비교
- **7. 화면 단위 기능정의**: 화면 ID(SC-01 형식) / 화면명 / 구성 요소 / 정책·조건·예외
- **8. 알림 정의**: No / 알림 유형 / 트리거 / 알림 문구 / 발송 조건 / 예외
- **9. 자동실행 / 동시실행 정책**: 기능·알림 중복 선택 가능 여부 매트릭스 + 설명
- **10. SLA (Service Level Agreement)**: No / 지표 / 목표값 / 측정 방법
- **11. CTQ (Critical To Quality)**: No / 품질 핵심 지표 / 기준 / 측정 방식
- **12. 예외 케이스 상세**: "~하더라도 ~인 경우" 형식으로 나열
- **13. Out of Scope**: 이번 버전 제외 범위 나열

### 필수여부(M/O) 표기 규칙

- **M (Mandatory)**: 반드시 포함하는 항목
- **O (Optional)**: 선택적으로 포함하는 항목
- 2.요구사항 체크리스트에서 각 항목에 M/O를 부여하고, 규칙 문구를 표 아래에 명시함

### 상품 구성(4번) 적응 규칙

- 통신사 멤버십 앱은 전용 디바이스가 없으므로 **4.1 디바이스 스펙**은 "해당 없음(앱 전용)"으로 명시
- **4.2 요금제**는 멤버십 등급·이용 조건(무료/유료 구분)으로 대체 작성

### SLA·CTQ 작성 원칙

- SLA·CTQ 지표는 **측정 가능한 수치(threshold)** 로 정의함 (예: "3초 이내", "99.9%", "정확도 90% 이상")
- 이 목표치는 이후 유저스토리 작성 시 스토리별 구현 태스크·수용 기준으로 구체화되는 **단일 출처(Source of Truth)** 역할을 함
- 모호한 표현("빠르게", "안전하게")은 금지함

## Word 서식 규칙

샘플 docx와 동일한 서식으로 재현함.

| 요소 | 규칙 |
|------|------|
| 글꼴 | 맑은 고딕, 본문 11pt |
| 제목·머리글·소제목 | 굵게(Bold), 크기는 본문과 동일(별도 확대 없음) |
| 페이지 | A4, 여백 위 약 3cm / 좌·우·아래 2.54cm(1인치) |
| 표 머리글 행 | 음영 `#F5F5F5` + 굵게 |
| 표 셀 테두리 | 실선 0.75pt, 색 `#E6E6E6` |
| 표 셀 정렬 | 내용 세로 가운데 정렬 |
| 하위 나열 | 글머리 기호(불릿) 목록 |
| 체크박스 | ■(선택) / □(미선택) 문자 |
| 킬러 폼(6번) 강조 | 기존/경쟁사와 다르거나 신규 추가된 항목은 굵게 강조 |
| 헤더·푸터 | "내부용(InternalUseOnly)" 보안 라벨은 재현하지 않음(비워 둠) |

### 생성·검증 방법

- `anthropic-skills:docx` 스킬을 사용하되, 위 서식 템플릿 docx를 기준으로 동일한 글꼴·여백·표 스타일을 적용
- 작성 완료 후 PDF로 변환해 육안으로 레이아웃이 샘플과 동일한지 확인

## 주의사항

- 각 섹션 내용은 이벤트스토밍·입력 자료를 근거로 도출하고, 근거 없는 신규 기능·정책 임의 추가 금지
- SLA·CTQ는 구체적 수치로 작성 (모호한 "빠르게", "안전하게" 표현 금지)
- 헤더·푸터의 내부용 보안 라벨(작성자 이메일·기밀 표시)은 재현 금지
- 기획 산출물이므로 정보기술(IT) 전문용어·약어는 설명 없이 사용 금지 (풀네임·쉬운 설명 병기)
- 6.정책 비교표(킬러 폼)는 기존/경쟁사와의 차별점을 한눈에 드러내는 핵심 섹션이므로 가장 공들여 작성

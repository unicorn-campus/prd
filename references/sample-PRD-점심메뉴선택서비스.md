## 2. 개요

### 2.1 배경

- 직장인·팀 단위 점심 결정에 평균 10~15분이 소요되며, 매일 반복되는 선택 피로(Decision Fatigue)가 발생함
- 팀 단위 점심의 경우 개인 취향 차이로 합의에 이르는 과정에서 갈등 비용이 발생함
- 기존 대안(지도 앱 검색, 단체 채팅방 투표)은 점심 결정에 특화되어 있지 않아 매번 수동으로 옵션을 취합해야 함

### 2.2 제품 한 줄 정의

- 개인/그룹의 점심 메뉴 결정을 30초 내로 단축시키는 추천·투표 서비스

### 2.3 목적

- 점심 메뉴 결정 시간과 갈등 비용을 줄이고, 반복 사용을 통해 개인화된 추천 품질을 높임

## 3. 문제 정의 및 목표

### 3.1 핵심 문제

| 문제 | 현재 대안 | 한계 |
|-----|---------|-----|
| 매일 반복되는 메뉴 결정 피로 | 개인 기억, 지도 앱 검색 | 취향 반영 없음, 검색 시간 소요 |
| 팀 단위 합의 어려움 | 단체 채팅방 자유 토론 | 발언권 편중, 결론 도출 지연 |
| 새로운 메뉴 발견 어려움 | 지인 추천, SNS 검색 | 현재 위치·시간·날씨 미반영 |

### 3.2 목표 (Goals)

- G1. 개인 점심 결정 소요 시간을 평균 10분 → 1분 이내로 단축
- G2. 그룹 점심 합의 소요 시간을 평균 15분 → 3분 이내로 단축
- G3. 재방문 사용자의 메뉴 추천 만족도(👍 비율) 70% 이상 확보

### 3.3 비목표 (Non-Goals)

- 배달/결제 대행은 다루지 않음 (Out of Scope, [5. 범위](#5-범위) 참조)
- 저녁·회식 메뉴 추천은 1차 버전 범위에서 제외

## 4. 타겟 사용자

### 4.1 페르소나

| 구분 | 페르소나 A | 페르소나 B |
|-----|-----------|-----------|
| 이름 | 김직장 (32세) | 이팀장 (41세) |
| 역할 | 사무직 개인 사용자 | 팀 점심 주최자 |
| 니즈 | 매일 뭘 먹을지 빨리 정하고 싶음 | 팀원 취향을 반영해 빠르게 합의하고 싶음 |
| 사용 맥락 | 점심시간 10분 전, 혼자 또는 동료 1~2명과 결정 | 5~8인 팀 점심, 그룹 채팅으로 조율 |
| 불편 | 매번 같은 곳만 가게 됨 | 의견 취합에 시간 소요, 소수 의견 묵살 |

### 4.2 Early Adopter 특성

- 사내 메신저(Slack, 카카오워크)를 업무에 상시 사용하는 5~50인 규모 스타트업/사무직 팀
- 점심 메뉴를 그룹 채팅으로 논의하는 습관이 있는 팀

## 5. 범위

### 5.1 In Scope (1차 버전)

- 개인용 랜덤/조건 기반 메뉴 추천
- 그룹 투표 생성 및 실시간 결과 집계
- 최근 방문 이력 기반 재방문 제외 옵션
- 위치·거리·영업시간 기반 필터링

### 5.2 Out of Scope (1차 버전 제외)

- 배달 주문 연동 및 결제
- 예약 기능
- 저녁·회식·야식 등 점심 외 시간대 추천
- 다국어 지원 (한국어 단일 언어로 시작)

---

## 6. 사용자 시나리오

### 시나리오 1. 개인 추천

1. 사용자가 앱 실행 후 "오늘 뭐 먹지" 버튼 탭
2. 현재 위치·시간·직전 방문 이력을 반영한 메뉴 3개 카드 노출
3. 마음에 드는 메뉴 선택 시 인근 식당 리스트 및 도보 시간 표시
4. 선택 결과가 방문 이력에 자동 반영되어 다음 추천에 활용됨

### 시나리오 2. 그룹 투표

1. 주최자가 "팀 투표 만들기" 선택 후 후보 메뉴 자동 추천(최근 팀 방문 이력 기반) 또는 직접 등록
2. 투표 링크를 팀 메신저(Slack/카카오워크)로 공유
3. 팀원이 링크 접속 후 로그인 없이 투표 참여(최대 3개 복수 선택 가능)
4. 마감 시간 도달 또는 전원 투표 완료 시 자동으로 결과 집계 및 확정 메뉴 공지

## 7. 기능 요구사항

우선순위는 MoSCoW 기준(M: Must, S: Should, C: Could, W: Won't)으로 표기함.

| ID | 기능 | 설명 | 우선순위 |
|----|-----|-----|---------|
| FR-01 | 개인 메뉴 추천 | 위치·시간·최근 방문 이력 기반 메뉴 3개 추천 | M |
| FR-02 | 재방문 제외 필터 | 최근 N일 내 방문한 식당/메뉴 추천에서 제외 | M |
| FR-03 | 그룹 투표 생성 | 후보 메뉴 등록 및 마감 시간 설정 | M |
| FR-04 | 그룹 투표 참여 | 링크 접속만으로 로그인 없이 투표 참여 | M |
| FR-05 | 투표 결과 자동 집계 | 마감 시 최다 득표 메뉴 자동 확정 및 공지 | M |
| FR-06 | 위치 기반 필터링 | 반경/도보 시간 기준 식당 필터링 | S |
| FR-07 | 카테고리/가격대 필터 | 한식·중식·양식 등, 가격대별 필터 | S |
| FR-08 | 메신저 공유 연동 | Slack/카카오워크 투표 링크 공유 | S |
| FR-09 | 방문 이력 관리 | 개인/팀 단위 방문 이력 조회 및 통계 | C |
| FR-10 | 날씨 기반 추천 조정 | 우천/폭염 시 실내 이동 가능한 메뉴 우선 노출 | C |

> 각 기능의 상세 시나리오·수용 기준(Acceptance Criteria)·Story Point는 `docs/plan/design/userstory.md`에서 UFR 단위로 구체화함.

## 8. 비기능 요구사항

| 구분 | 요구사항 |
|-----|---------|
| 성능 | 추천 결과 응답 2초 이내, 투표 결과 집계 1초 이내 |
| 가용성 | 점심시간대(11:00~13:30) 가용성 99.9% 이상 |
| 확장성 | 동시 투표 세션 1,000개 이상 처리 |
| 보안 | 투표 참여 시 개인정보 미수집(닉네임만 사용), 링크는 추측 불가능한 토큰 기반 |
| 접근성 | 로그인 없는 투표 참여 플로우는 모바일 웹 기준 3탭 이내 완료 |
| 데이터 보존 | 방문 이력은 최근 90일까지 보관 후 자동 삭제 |

## 9. UX 원칙

- 점심시간이라는 짧은 의사결정 창구를 고려해 모든 핵심 액션은 3탭 이내로 완료
- 그룹 투표는 참여자의 로그인 부담을 제거해 참여율을 극대화
- 추천 결과는 "왜 이 메뉴인지"에 대한 근거(최근 미방문, 거리, 날씨 등)를 함께 노출

## 10. 성공 지표

| 지표 | 정의 | 목표치 |
|-----|-----|-------|
| 결정 소요 시간 | 앱 실행~메뉴 확정까지 걸린 시간 | 개인 1분 이내, 그룹 3분 이내 |
| 투표 참여율 | 투표 생성 대비 참여 완료 비율 | 80% 이상 |
| 재방문율 (Retention) | 7일 내 재사용 비율 | 40% 이상 |
| 추천 만족도 | 추천 결과에 대한 👍 비율 | 70% 이상 |
| 바이럴 계수 | 투표 링크를 통한 신규 유입 비율 | 1인당 0.3명 이상 |

## 12. 리스크 및 가정

### 12.1 가정

- 팀 점심 결정의 주 채널이 메신저 그룹 채팅이라는 전제
- 사용자가 위치 정보 제공에 동의한다는 전제

### 12.2 리스크

| 리스크 | 영향 | 대응 방안 |
|-------|-----|---------|
| 식당 정보(영업시간·휴무) 정확도 저하 | 잘못된 추천으로 신뢰도 하락 | 지도 API(카카오맵/구글맵) 실시간 연동, 사용자 신고 기능 |
| 초기 방문 이력 데이터 부족 (Cold Start) | 추천 품질 저하 | 위치·시간 기반 규칙 추천으로 보완 후 이력 누적에 따라 개인화 강화 |
| 그룹 투표 저조 참여 | 핵심 가치 제안 실패 | 로그인 없는 참여 플로우, 메신저 딥링크 최적화 |

## 13. 의존성

- 지도/장소 API: 카카오맵(국내), 구글맵(예비)
- 메신저 공유: Slack Webhook, 카카오워크 공유 API
- 날씨 API: 기상청 또는 OpenWeatherMap (v1.0 범위)

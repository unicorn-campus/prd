## 범례(스티키노트 색상)

| 색상 | 의미 |
| --- | --- |
| 파랑 | Command(명령) |
| 주황 | Domain Event(도메인 이벤트) |
| 보라 | Policy(정책/업무 규칙) |
| 초록 | External System(외부 시스템/액터) |
| 노랑 | Actor(행위 주체) |
| 빨강/초록 헤더 + 흰 바탕 | Data Field(조회·전달 데이터 항목) |

용어 설명:  
- CDC(Change Data Capture, 데이터 변경 감지): 원부 DB의 데이터 변경 내용을 감지해 다른 저장소에 반영하는 방식  
- Publish/Subscription(발행/구독): 이벤트 발생 시 구독 중인 시스템에 알려 후속 처리를 트리거하는 연동 방식  
- nMIMO: 문자(SMS) 발송 시스템명

이벤트스토밍 보드는 아래 6개 섹션(클러스터)으로 구성됨: 멤버십 가입자 정보, 멤버십 상품, 포인트 정보, 혜택 정보, 고객 알림서비스, 상담

---

## 1. 멤버십 가입자 정보

### 1-1. 멤버십 사용내역 조회

- 액터: 멤버십 가입자(온라인), 가맹점(오프라인)
- 외부 시스템: 멤버십 웹/앱
- Command: 멤버십 사용내역을 조회한다.
- Event: 멤버십 조회 요청을 받았다.
- Data Field: 제휴사코드, 멤버십코드, 혜택코드, 가맹점
- Command: 고객 정보인증을 재확인한다.
- Policy: 멤버십 유효성을 확인한다. (멤버십코드 / 제휴사코드 / 혜택코드)

### 1-2. 멤버십 사용 요청 → 잔여포인트 조회

- 액터: 멤버십 가입자(온라인), 가맹점(오프라인)
- 외부 시스템: 멤버십 웹/앱
- Command: 멤버십 사용을 요청한다.
- Event: 멤버십 사용 요청을 받았다.
- Data Field: 제휴사코드, 멤버십코드, 혜택코드, 가맹금액
- Command: 고객 정보인증을 재확인한다.
- Policy: 멤버십 유효성을 확인한다. (멤버십코드 / 제휴사코드 / 혜택코드)
- Data Field: 제휴사코드, 멤버십코드, 혜택코드, 가맹금액, 잔여포인트, 사용가능여부(잔여포인트·고객등급으로 사용가능 여부 확인)
- Command: 고객의 멤버십 등급 및 잔여포인트를 캐시(Redis)에서 요청한다.
- Event: 레디스를 조회해서 데이터를 찾았다.
- Data Field: 제휴사코드, 멤버십코드, 혜택코드, 고객등급, 사용포인트, 잔여포인트, 사용자캐시 존재/조회여부(잔여포인트·고객등급으로 사용가능 여부 확인)
- Policy: 멤버십 등급을 확인한다(해당 멤버십 고객이 혜택 제공 가능한 등급인지 확인). 잔여포인트를 확인한다(고객이 혜택 제공을 받을 포인트가 있는지 확인). 승인 이력 여부를 확인한다.
- 분기 처리:
  - Event: 캐시에서 조회가 되지 않아 '조회 DB'에서 데이터를 조회했다.
  - Event: 캐시에 데이터를 찾아서 조회된 데이터를 반환했다.

### 1-3. 멤버십 결제 취소 요청

- 액터: 멤버십 가입자(온라인), 제휴사(오프라인), 상담사
- 외부 시스템: 제휴사 포스기, 상담사운영시스템
- Command: 멤버십 결제 취소 요청을 보낸다.
- Event: 멤버십 결제 취소 요청을 받았다.
- Data Field: 제휴사코드, 멤버십코드, 혜택코드, 사용 처리시퀀스
- Command: 고객 정보인증을 재확인한다.

### 1-4. 멤버십 Core 원부 DB 조회(캐시 미스 시 후속 처리)

- (1-2, 1-3에서 이어짐) Event: 캐시에서 조회가 되지 않아 '조회 DB'에서 데이터를 조회했다. / 캐시에 데이터를 찾아서 조회된 데이터를 반환했다.
- Command: 멤버십 core 원부 DB에 멤버십 사용 관련 조회요청을 보낸다.
- Event: 멤버십 Core 원부 데이터 정보가 조회되었다.(멤버십 등급, 멤버십 사용이력)
- Data Field: 제휴사코드, 멤버십코드, 혜택코드, 고객등급, 사용포인트, 사용자캐시존재여부, 가맹금액, 고객등급, 잔여포인트, 사용가능여부

## 2. 멤버십 상품

### 2-1. 멤버십 정보 조회 후처리

- Command: 멤버십 정보 조회 후처리 요청을 한다.
- Event: 잔여포인트 및 멤버십 등급 등 멤버십 정보가 조회되었다.
- Data Field: 멤버십코드, 고객등급, 잔여포인트, 멤버십사용이력

### 2-2. 멤버십 사용 후처리 → 포인트/혜택 갱신(사용)

- Command: 멤버십 사용 후처리 요청을 한다.
- Event: 멤버십 사용이 가능한 상태였다.
- Data Field: 사용처리결과(성공/실패), 사용처리포인트
- Policy: 가용상태 = 잔여포인트 > 사용할 포인트
- 병렬 처리(Publish/Subscription):
  - Command: 멤버십 혜택 갱신 요청을 보낸다.
    - Event: 멤버십 혜택이 멤버십 Core 원부 데이터 Command DB에서 계산되었다.
    - Data Field: 사용처리결과(성공/실패), 사용처리포인트
    - Policy: 해당 멤버십 고객의 요청만 계산한다.
  - Command: 멤버십 포인트 감소 요청을 보낸다.
    - Event: 멤버십 포인트가 멤버십 Core 원부 데이터 Command DB에서 차감되었다.
    - Data Field: 사용처리결과(성공/실패), 사용처리포인트
    - Policy: 해당 멤버십 고객의 요청만 찾아서 차감한다.
- Event: CDC 솔루션에서, 멤버십 Core 원부 데이터에 대한 Command DB 내 포인트 사용 관련 변동 내용을 식별하였다.
- Command: 멤버십 Core 원부 데이터 갱신 요청을 한다.
- Event: 멤버십 Core 원부 데이터에 대한 조회 DB가 현행화 되었다.

### 2-3. 멤버십 취소 후처리 → 포인트/혜택 갱신(취소)

- Command: 멤버십 취소 후처리 요청을 한다.
- Event: 멤버십 취소가 가능한 상태였다.
- Data Field: 사용 처리이력, 취소 처리포인트
- Policy: 멤버십 사용 이력이 존재한다.
- 병렬 처리(Publish/Subscription):
  - Command: 멤버십 혜택 갱신 요청을 보낸다.
    - Event: 멤버십 혜택이 멤버십 Core 원부 데이터 Command DB에서 계산되었다.
    - Data Field: 취소 처리결과(성공/실패), 취소 처리포인트
  - Command: 멤버십 포인트 증가 요청을 보낸다.
    - Event: 멤버십 포인트가 멤버십 Core 원부 데이터 Command DB에서 증가되었다.
    - Data Field: 취소 처리결과(성공/실패), 취소 처리포인트
- Event: CDC 솔루션에서, 멤버십 Core 원부 데이터에 대한 Command DB 내 포인트 취소 관련 변동 내용을 식별하였다.
- Command: 멤버십 Core 원부 데이터 갱신 요청을 한다.
- Event: 멤버십 Core 원부 데이터에 대한 조회 DB가 현행화 되었다.

(2-2, 2-3의 흐름은 각각 3번 섹션 "포인트 정보", 4번 섹션 "혜택 정보"와 동일 패턴이 반복되는 구조임 — 사용/취소 각각에 대해 포인트·혜택이 함께 갱신됨)

## 3. 포인트 정보 / 4. 혜택 정보

- 2-2, 2-3과 동일한 사용·취소 흐름이 포인트 정보·혜택 정보 관점에서 반복 구성됨
- 최종적으로 두 갱신 이벤트(포인트 사용 관련 변동, 포인트 취소 관련 변동) 모두 "멤버십 Core 원부 데이터에 대한 조회 DB가 현행화 되었다."로 수렴
- 이 결과 이벤트가 5번 섹션 "고객 알림서비스"로 연결됨

## 5. 고객 알림서비스

- (2-2, 2-3, 상담 섹션의 포인트 증가/감소 이벤트로부터 연결됨)
- Command: 포인트 내역 문자발송
- Event: 고객에게 포인트 내역 문자가 발송되었다.
- 외부 시스템: 문자발송시스템(nMIMO)

## 6. 상담

### 6-1. 전화 상담 접수

- 액터: 상담사
- Command: 전화 수신
- Event: 고객으로부터 상담사에게 전화 연락이 왔다.
- Policy: 본인 여부를 확인한다.

### 6-2. 고객 포인트 내역 조회

- Command: 고객 포인트 내역 조회
- Event: 상담시스템에서 고객의 포인트 사용 내역이 조회되었다.
- Policy: 상담 요청 온 본인의 정보만 조회한다.

### 6-3. 상담사 처리 유형(4가지) → 잔여포인트 증가/감소

- Command: 고객 포인트 사용 내역 취소 → Event: 상담사로부터 고객의 포인트 사용 내역이 취소되었다.
- Command: 고객 포인트 사용 내역 추가 → Event: 상담사로부터 고객의 포인트가 추가되었다.
- Command: 고객 포인트 사용 내역 추가 → Event: 상담사로부터 고객의 포인트 사용 내역이 추가되었다.
- Command: 고객 포인트 적립 취소 → Event: 상담사로부터 고객의 적립 포인트가 회수되었다.

집계 결과:
- (취소 + 추가) → Command: 고객 잔여 포인트 증가 → Event: 고객의 잔여포인트가 증가되었다.
- (추가 + 적립취소) → Command: 고객 잔여 포인트 감소 → Event: 고객의 잔여포인트가 감소되었다.
- 두 결과 모두 5번 섹션 "고객 알림서비스"(포인트 내역 문자발송)로 연결됨

## 전체 흐름 요약

1. 멤버십 가입자(온라인) 또는 가맹점(오프라인)이 멤버십 조회/사용/결제취소를 요청함
2. 캐시(Redis) 우선 조회 → 미스 시 멤버십 Core 원부 DB 조회로 폴백함
3. 멤버십 등급·잔여포인트 기준으로 사용 가능 여부를 정책으로 검증함
4. 사용/취소 확정 시 멤버십 혜택·포인트가 Publish/Subscription 방식으로 병렬 갱신됨
5. CDC로 Command DB 변경을 감지해 조회 DB를 현행화함(최종 일관성 확보)
6. 상담사도 별도 채널로 고객 포인트 사용내역 취소/추가, 적립취소를 처리하여 잔여포인트를 증감시킴
7. 포인트 변동이 발생하면 고객에게 문자(nMIMO)로 내역을 안내함

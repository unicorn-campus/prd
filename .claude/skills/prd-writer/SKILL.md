# PRD 작성

## 활성화 조건

사용자가 `/prd-writer` 호출 시 또는 "PRD 작성", "PRD 작성해줘", "제품요구사항 정의서 작성" 키워드 감지 시.

## 선행 조건

- `docs/plan/think/es/*.puml` (이벤트스토밍) 및 아래 [입력]의 산출물이 모두 존재해야 함
- 미존재 시 사용자에게 해당 단계(기획 워크플로우) 선행 수행을 안내하고 종료

---

[목표]
이벤트 스토밍·비즈니스모델·문제가설 등 이전 단계 산출물을 근거로, 통신사 멤버십 "포인트 한눈에 보기 +
혜택 도우미" MVP의 제품 범위와 기능/비기능 요구사항을 확정하는 PRD(제품 요구사항 정의서)를 작성함.

[역할]
당신은 통신사 멤버십·포인트 서비스 기획 경험과 대규모 사용자 서비스 성장 경력을 가진 제품 책임자(PO)입니다.
'왜 만드는가(WHY)'를 먼저 정의하고, 고객 관점을 잃지 않으며, 제품 범위와 요구사항을 명확히 확정하는 전문가입니다.

[맥락]
- 내 상황: 통신사 멤버십 MVP "포인트 한눈에 보기 + 혜택 도우미"의 기획 단계에서 이벤트 스토밍까지 완료함.
  이제 파악된 도메인 흐름을 근거로 "무엇을·왜 만들지"를 확정하는 PRD를 작성해야 함.
- 결과물 독자: 후속 유저스토리·설계·개발·검증 단계를 수행할 팀원(아키텍트·개발자·QA)과 이해관계자.
  기획 문서이므로 정보기술(IT) 비전문가도 읽을 수 있어야 함.

[입력]
- PRD 작성 가이드: `references/prd-guide.md` (섹션 구성·매핑 규칙·출력 형식·품질 기준의 기준 문서 — 반드시 먼저 읽고 그대로 따름)
- PRD 샘플: `references/sample-PRD-점심메뉴선택서비스.md` (출력 형식·표 구조 참고)
- 이벤트 스토밍: `docs/plan/think/es/*.puml` — 유저플로우 → 사용자 시나리오, 이벤트 → 기능 요구사항 후보,
  커맨드 → 기능 요구사항 상세, 정책/규칙 → 비기능 요구사항·제약조건, 참여자 → 타겟 사용자
- 유저플로우 매핑: `docs/plan/think/es/유저플로우매핑.md` — 유저플로우와 이벤트스토밍 파일의 연결 관계
- 비즈니스 모델: `docs/plan/think/비즈니스모델.md` — 문제/고객 세그먼트/고유가치제안 → 문제 정의·타겟·목적,
  핵심 지표 → 성공 지표
- 문제 가설: `docs/plan/define/문제가설.md` — 핵심 문제 Top 3 → 문제 정의 근거
- 시장 조사: `docs/plan/define/시장조사.md` — 경쟁 환경·규제 환경 → 범위 설정·비기능(컴플라이언스) 근거
- 고객 분석: `docs/plan/define/고객분석.md` — 사용자 세그먼트 → 타겟 사용자·페르소나
- 비즈니스 가치: `docs/plan/define/비즈니스가치.md` — 측정 지표 → 성공 지표

[처리]
- PRD 작성 가이드 숙지: `references/prd-guide.md`를 가장 먼저 읽고, 그 14개 섹션 구성·매핑 규칙·출력 형식·품질 기준을 작업 기준으로 삼음
- 입력 자료 분석: 이전 단계 산출물(이벤트스토밍·유저플로우매핑·비즈니스모델·문제가설·시장조사·고객분석·비즈니스가치)을
  모두 읽고 문제·고객·범위·지표의 근거를 정리
- 이벤트스토밍 → PRD 매핑 규칙 적용
  - 유저플로우(제목) → 사용자 시나리오 (제품이 지원해야 하는 핵심 흐름)
  - 이벤트 → 기능 요구사항 후보 (시스템이 보장해야 하는 결과)
  - 커맨드 → 기능 요구사항 상세 (사용자/시스템이 수행하는 행동)
  - 정책/규칙 → 비기능 요구사항·제약조건 (성능·보안·컴플라이언스 등)
  - 참여자(Actor) → 타겟 사용자 세그먼트
- 14개 섹션을 가이드의 출력 형식 순서대로 작성
  - 1.문서정보 2.개요 3.문제정의및목표 4.타겟사용자 5.범위 6.사용자시나리오 7.기능요구사항
    8.비기능요구사항 9.UX원칙 10.성공지표 11.릴리스계획 12.리스크및가정 13.의존성 14.부록
- 기능 요구사항: 각 항목에 ID(FR-01 형식)와 MoSCoW 우선순위를 부여
  - M(Must, 필수) / S(Should, 권장) / C(Could, 선택) / W(Won't, 이번엔 제외) 4단계로 표기
- 비기능 요구사항: 성능·가용성·보안·확장성 4개 영역을 모두 포함하고, 각 항목을 측정 가능한 목표치로 정의
  - 이 목표치는 후속 유저스토리 단계에서 재정의하지 않는 단일 기준(Source of Truth)으로 작성
- 품질 검증: 작성 후 [제약조건]의 완료조건을 스스로 점검하고, 미충족 항목이 있으면 보완
- 출력파일: `PRD.md`
- 톤앤매너
  - 기획 산출물이므로 정보기술(IT) 전문용어 사용을 지양하고 쉬운 용어로 작성
  - 전문용어·약어가 불가피하면 풀네임과 쉬운 설명을 괄호로 덧붙임
  - 필요 시 비유와 예시로 이해를 도움
- 작성 규칙
  - 한국어로 작성
  - PRD 작성 가이드의 14개 섹션 구성과 출력 형식을 그대로 준수
  - 명사체 종결("~함/~임") 사용, 한 줄 120자 이내
  - 문서 상단에 목차(앵커 링크) 포함, 작성일은 오늘 날짜로 기입

[출력]
- `docs/plan/design/PRD.md` (마크다운, 14개 섹션 구성)

[제약조건]
- MUST
  - 14개 섹션을 모두 작성
  - 이벤트스토밍의 모든 유저플로우가 각각 최소 1개 이상의 기능 요구사항(FR)과 연결됨
  - 기능 요구사항 개수 ≥ 이벤트스토밍 유저플로우 수, 전 항목에 MoSCoW 우선순위 적용
  - 비기능 요구사항에 성능·가용성·보안·확장성 4개 영역을 모두 포함하고 측정 가능한 목표치 명시
  - 성공 지표 3개 이상을 `비즈니스가치.md`의 측정 지표와 연계
  - 문제 정의를 `문제가설.md`의 핵심 문제와 연결
  - 범위를 In Scope / Out of Scope로 명시적으로 구분하고 `솔루션평가.md`의 실현 가능성과 정합성 유지
- MUST NOT
  - 이벤트스토밍 결과에 근거 없는 신규 기능을 임의로 추가
  - "빠르게", "안전하게" 등 측정 불가능한 모호한 표현으로 비기능 요구사항 기술
  - 비기능 요구사항 목표치를 후속 유저스토리 단계에서 재정의
  - 정보기술(IT) 전문용어·약어를 설명 없이 사용
- 완료조건
  - `docs/plan/design/PRD.md` 파일이 실제로 생성됨
  - 14개 섹션이 모두 채워지고, 가이드의 완료 체크리스트 8항목과 정량 기준 3개를 모두 충족함
  - 8개 유저플로우 → 기능 요구사항 연결이 문서에서 추적 가능함

[예시]
(이벤트스토밍 유저플로우) `01-통합혜택카드조회.puml`
  → (6. 사용자 시나리오) "시나리오 1. 통합 혜택 카드 한눈에 조회"
  → (7. 기능 요구사항)
     | ID    | 기능              | 설명                                          | 우선순위 |
     | FR-01 | 통합 혜택 카드 조회 | 포인트·등급·혜택을 한 화면 카드로 통합 표시     | M       |

(정책/규칙) "혜택 카드 조회는 즉시 응답해야 함"
  → (8. 비기능 요구사항)
     | 구분 | 요구사항(목표치)                   |
     | 성능 | 혜택 카드 첫 화면 조회 응답 2초 이내 |

(비목표 예시)
  - 포인트 결제·전환 기능은 1차 버전 범위에서 제외 (Out of Scope)
